# FASE 2 — Assembly (gcc -S)
# Cole aqui 5 a 10 linhas de soma.s que você considera mais
# interessantes e explique o que cada uma faz.

Trecho de soma.s:
addi sp,sp,-32       
sd   s0,24(sp)       
li   a1,5            
li   a0,7            
call soma            
call printf          
jr   ra

Explicação: 
addi sp,sp,-32 — reserva 32 bytes na pilha para a função
sd s0,24(sp) — salva o registrador s0 na pilha (preserva o contexto)
li a1,5 — carrega o valor 5 no registrador a1 (segundo argumento de soma)
li a0,7 — carrega o valor 7 no registrador a0 (primeiro argumento de soma)
call soma — chama a função soma
call printf — chama o printf para imprimir o resultado
jr ra — retorna da função (jump para o endereço de retorno)


# FASE 3 — Código objeto (gcc -c)
# Cole a saída de: file soma.o

file soma.o:

soma.o: ELF 64-bit LSB relocatable, UCB RISC-V, version 1 (SYSV), not stripped

nm soma.o:

0000000000000000 r .LC0
0000000000000028 T main
                 U printf
0000000000000000 T soma

Observação sobre nm:
T soma — a função soma está definida neste arquivo (T = Text, seção de código)
T main — a função main também está definida aqui
U printf — o printf está indefinido (U = Undefined), ou seja, foi usado no código mas não foi definido aqui — é da biblioteca padrão do C. O ligador será responsável por encontrar e conectar essa função na próxima fase
r .LC0 — é a string "%d\n" guardada na seção de dados somente leitura

# FASE 4 — Ligação (gcc soma.o -o soma)
# O que mudou após a ligação? Execute ./soma e registre a saída.

Saída de ./soma: 12
Após a ligação, o ligador resolveu o símbolo U printf buscando ele na biblioteca padrão do C (libc), e gerou o executável final soma que rodou corretamente.


# QUESTÕES DE SÍNTESE

# 1. O nm mostrou printf como "U" (undefined). Em qual slide da
#    apresentação esse mesmo problema apareceu? O que o ligador
#    faz com esse símbolo?

Resposta 1: O ligador pega o símbolo printf marcado como U (Undefined) no soma.o e encontra a definição dele dentro da biblioteca padrão do C (libc), resolvendo a referência e tornando o programa executável completo.


# 2. Compare soma.i com soma.s: qual das duas saídas é análoga
#    ao "código objeto" do montador histórico? Justifique.

Resposta 2: O arquivo soma.s (assembly) é o análogo ao "código objeto" do montador histórico, pois é a saída do compilador em linguagem de baixo nível antes de virar binário. Porém, o arquivo que o computador realmente executa é o soma — o executável final gerado na Fase 4, após o ligador resolver todos os símbolos.
